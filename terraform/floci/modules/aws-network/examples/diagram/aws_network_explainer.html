Senior Infra Refresher

Multi‑AZ VPC, public edge, private compute.

This is a classic production network: a regional VPC split across two Availability Zones, internet ingress through an Application Load Balancer in public subnets, workloads in private subnets, zonal NAT for outbound internet, and an S3 Gateway Endpoint to keep S3 traffic off the public internet.

Mental model in one sentence

  • Public subnets host internet-facing infrastructure: ALB nodes and NAT Gateways.
  • Private subnets host application servers/ASG instances with no public IPs.
  • Route tables decide whether packets go to IGW, NAT, local VPC, or S3 endpoint.
  • Security groups are stateful; NACLs are stateless subnet guardrails.
01 / Architecture

What each component is doing

VPC

10.0.0.0/16 boundary

One isolated regional network. All subnets route locally to each other by default through the implicit local route.

AZ A

10.0.1.0/24 + 10.0.11.0/24

Public subnet A contains ALB node A and NAT‑A. Private subnet A contains app/server A.

AZ B

10.0.2.0/24 + 10.0.12.0/24

Same pattern in another failure domain. This gives zonal resilience and avoids cross‑AZ NAT dependency during normal operation.

IGW

Internet Gateway

Attached to the VPC. Public route tables send 0.0.0.0/0 → igw. Required for public ALB ingress and NAT Gateway egress.

NAT

Managed egress

Each NAT Gateway sits in a public subnet with an Elastic IP. Private subnets send default outbound traffic to their same‑AZ NAT.

S3 Endpoint

Private AWS service path

Gateway endpoint adds prefix-list routes for S3 in private route tables, so S3 traffic bypasses NAT/IGW and can be restricted by endpoint policy.

02 / Packet Walks

How traffic flows

Internet user → application

Client resolves ALB DNS. Packet enters AWS edge, hits the VPC through the Internet Gateway path, then lands on one of the ALB nodes in public subnet A or B. The ALB forwards to healthy private targets over their private IPs. Response returns app → ALB → client.

client → IGW → public ALB subnet → ALB node → private app subnet → server

Private server → internet

Server A has no public IP. Its route table sends 0.0.0.0/0 to NAT‑A. NAT‑A rewrites the source to its Elastic IP and sends traffic through the public subnet route table to the IGW. Return traffic is mapped back by NAT state.

server A → private-rt-a → NAT-A/EIP → public-rt-a → IGW → internet

Private server → S3

The S3 prefix-list route is more specific than 0.0.0.0/0, so it wins. Traffic goes to the S3 Gateway Endpoint, not NAT. Benefits: lower cost, no public internet path, IAM/endpoint policy controls.

server → private route table → com.amazonaws.region.s3 endpoint → S3

AZ failure behavior

ALB stops routing to unhealthy targets/AZs. ASG can replace instances in healthy subnets. With one NAT per AZ, surviving AZ egress continues. If NAT‑A alone fails, subnet A egress fails unless you automate failover to NAT‑B, accepting cross‑AZ data charges.

Public vs private is a route-table property.

A subnet is “public” when it has a default route to an IGW and resources have public IPs. A subnet is “private” when default internet egress goes through NAT or not at all.

Most production bugs are route + SG mismatches.

Debug with Reachability Analyzer, VPC Flow Logs, route table effective associations, NACL ephemeral ports, ALB target health, and NAT metrics.

04 / Review Notes

Senior-level points to say out loud

Resilience
  • Two AZs isolate failures.
  • ALB spans public subnets and health-checks targets.
  • ASG spans private subnets.
  • NAT per AZ avoids a single egress bottleneck.
Security
  • Apps have no public IPs.
  • Inbound is only ALB → app SG.
  • NACLs are coarse stateless guardrails.
  • S3 endpoint policy can restrict buckets/actions.
Cost/ops
  • NAT hourly + per-GB charges are material.
  • S3 Gateway Endpoint saves NAT data processing.
  • Cross-AZ NAT failover trades availability for cost.
  • Flow Logs help prove packet direction.
